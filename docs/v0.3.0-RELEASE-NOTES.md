# Vortix v0.3.0 â€” what's new

**v0.3.0 is the architectural migration v1 ship + the deferred-subsystems bundle.** Existing CLI commands are unchanged. Three new top-level subcommands (`secrets`, `audit`, `daemon`) earn their slots â€” each a real new noun with no pre-existing equivalent. CI integration tests against real `wg-quick` + `openvpn` ship as a netns harness. Daemon authentication posture documented in `SECURITY.md`; engine routing through the daemon ships as v0.3.x hardening. **Lifecycle hooks are deferred to v0.3.x** â€” the brainstorm and architectural sketches stay on file, but the implementation rolls back to give the UX another design iteration.

## Highlights

- **Engine FSM + session journal.** Internal connection state is now a
  typed state machine. Every session writes a JSONL event log to
  `${XDG_DATA_HOME}/vortix/sessions/*.jsonl` with 30-day / 30-file
  retention. The session path is surfaced via `vortix info`; you tail
  it with `tail -f` + `jq`.
- ~~**Lifecycle hooks**~~ â€” deferred to v0.3.x. The architectural
  groundwork (journal broadcast bus, schema brainstorm, TUI surface
  design) stays on file; the implementation lands in a future
  iteration after another UX pass.
- **`vortix audit`** (issues [#168](https://github.com/Harry-kp/vortix/issues/168),
  [#166](https://github.com/Harry-kp/vortix/issues/166)). Per-process
  socket snapshot â€” `vortix audit --pid <N>` filters to one process,
  `--vpn-only` to sockets on the tunnel interface, `--json` for
  structured envelope. Linux (`/proc/net`) + macOS (`lsof`) impls;
  Windows stub returns `PlatformUnsupported`.
- **`vortix daemon`** (issue [#16](https://github.com/Harry-kp/vortix/issues/16)).
  Long-running IPC server skeleton. Unix socket with mode 0600
  permissions. Length-prefixed JSON framing. Engine routing through
  the daemon completes in v0.3.x (the contract + wire are ready
  today; the dispatch routes engine ops in the next hardening
  iteration).
- **CI integration tests** (issue [#162](https://github.com/Harry-kp/vortix/issues/162)).
  New workflow that spins up a privileged Ubuntu Docker container,
  creates two network namespaces, brings up a WireGuard tunnel
  between them, and exercises `vortix up/status/down` + killswitch
  engage/release end-to-end.
- **Layered secret store.** Optional encrypted store backed by the OS
  keyring (Keychain on macOS, Secret Service on Linux) with an
  AES-256-GCM + argon2id on-disk fallback. Manage entries with
  `vortix secrets {set,get,delete}`. Existing `.auth` files keep
  working â€” the store is opt-in.
- **`vortix show --raw --inline-secrets`.** Streams the profile config
  to stdout with stored credentials appended as a
  `# vortix-secret:<base64>` trailing comment. For sharing a profile
  with a teammate who needs the credentials inline.
- **Hardened startup.** Defense-in-depth on the profile sidecar
  migration (`VORTIX_SKIP_MIGRATION=1` escape hatch), warn-only
  orphan-daemon scan when a previous run left `wg-quick` / `openvpn`
  behind, and a CI ceiling on `vortix --version` cold-start time so
  future changes can't inflate it silently.
- **Cargo workspace + 12 internal crates.** The codebase is now a flat
  matklad-style workspace with capability ports (Tunnel, Killswitch,
  DNS, Interface, NetworkStats, RouteTable, CommandRunner) in
  `vortix-core` and per-OS / per-protocol impls behind them. Adding
  new protocols or platforms is now mechanical. See
  [architecture-migration-v1.md](architecture-migration-v1.md).
- **JSON output carries `schema_version`.** Every `--json` envelope
  now begins with `"schema_version": 1`. The contract is versioned
  from day one.

## What's automatic

- **Profile sidecar backfill.** A `<name>.meta.toml` appears next to
  each existing `.conf` / `.ovpn` at first launch. Idempotent â€”
  re-runs are no-ops. v0.2.x ignores these files, so rollback is
  safe.
- **Existing CLI commands.** `vortix up`, `down`, `status`, `list`,
  `import`, `show`, `delete`, `rename`, `reconnect`, `killswitch`,
  `release-killswitch`, `info`, `update`, `report`, `completions` all
  behave exactly as before.
- **Killswitch state.** Pf / iptables / nftables rules persist across
  the binary upgrade. Active VPN sessions also survive the upgrade â€”
  vortix doesn't supervise its spawned `wg-quick` / `openvpn`
  daemons.

## What's opt-in

- **Encrypted secret store.** Empty by default. Move OpenVPN
  credentials into it with
  `echo -n 'user:pass' | vortix secrets set creds/<profile>`. See
  [MIGRATION.md](MIGRATION.md#encrypted-secret-store-opt-in).
- **`settings.toml`.** Figment-layered (defaults â†’ system â†’ user â†’
  env). You don't need to create one; the runtime defaults match
  v0.2.x behavior. Inspect what's resolved by editing your
  `settings.toml` directly.
- **Session journal on-disk persistence.** Default on. To disable
  disk persistence, add `[journal] disk = false` to `settings.toml` â€”
  the in-process broadcast still works.

## What's NOT in v0.3.0 (deferred to v0.3.x or later)

- **No Windows binary.** A `vortix-platform-windows` stub crate
  compiles on Windows but every port returns `PlatformUnsupported`.
  Issue [#17](https://github.com/Harry-kp/vortix/issues/17).
- **`vortix daemon` ships skeleton-only.** Architecture, wire
  contract, socket binding, and frame parser are in place; engine
  routing through the daemon completes in v0.3.x. Until then,
  `vortix daemon` accepts client connections but returns
  `IpcError::Internal("engine wiring not yet connected")` for
  `Execute`/`Snapshot`/`Subscribe`. See [SECURITY.md](../SECURITY.md#daemon-authentication-model-v030-phase-e)
  for the v0.3.0 auth posture + the deferred-hardening list.
- **No "run without sudo."** Privilege separation builds on the
  daemon engine routing; `SO_PEERCRED` enforcement + read-only-ops-
  bypass land in v0.3.x. Issue [#153](https://github.com/Harry-kp/vortix/issues/153).
- **No macOS CI integration parity.** v0.3.0's integration workflow
  is Ubuntu-only (GH Actions macOS runners don't support
  network-namespace + sandboxed `wg-quick` cleanly). Documented as
  follow-up.
- **No standalone `engine` / `journal` / `migrate` / `settings` / `export`
  subcommands.** The underlying architecture is all there, but these
  surfaces didn't earn top-level CLI real estate. The journal path
  surfaces via `vortix info`, the secret-inlining export is now a
  `--inline-secrets` flag on `vortix show --raw`, the migration runs
  at startup, and figment settings are inspectable by reading your
  `settings.toml`.

## Upgrade and rollback

Read [MIGRATION.md](MIGRATION.md) for upgrade instructions. Rollback
in one command:

```sh
cargo install vortix --version 0.2.2 --force
```

Migration artifacts (`.meta.toml` sidecars, `secrets.enc`,
`sessions/*.jsonl`) are inert to v0.2.x.
